## 三、上线阻塞项（P0 - 必须解决）

> 以下问题不解决，不应上线到生产环境。

### P0-1：短信验证码实际发送
**问题：** 客户端注册/登录依赖手机验证码，后端逻辑已完整（含频率限制、TTL、尝试次数），但验证码仅存储在内存中，未接入实际短信服务商发送。开发环境使用固定码 `123456`。

**影响：** 生产环境客户无法收到验证码，注册/登录完全无法正常使用。

**解决方案：** 接入阿里云短信 / 腾讯云短信 SDK，在 `VerificationCodeService.generate()` 后增加短信发送调用。预计工作量：4-8 小时。

---

### P0-2：生产环境配置管理
**问题：** 缺乏完整的生产环境配置文档和安全配置核查。

**影响清单：**
- JWT Secret 是否已更换为随机强密钥
- `ALLOW_DEV_MVP_VERIFICATION_CODE` 是否在生产中关闭
- 数据库连接串是否安全配置
- CORS 配置是否精确到域名（当前 `origin: '*'`）
- `UPLOAD_BASE_URL` 是否指向正确的生产域名

**解决方案：** 创建 `.env.production` 模板，逐项核查，严格禁用所有开发用旁路。预计工作量：2-4 小时。

---

### P0-3：文件存储（图片）生产方案
**问题：** 当前图片上传存储至本地磁盘（`/uploads/`），Docker 容器重启后文件丢失。

**影响：** 作品图片、头像、设计图全部会在容器重启时丢失。

**解决方案：** 接入对象存储（阿里云 OSS / 腾讯云 COS / AWS S3），将上传逻辑的本地写入替换为 SDK 上传。预计工作量：1-2 天。

---

### P0-4：数据库持久化与备份策略
**问题：** 当前使用 SQLite（`dev.db`），不适合生产高并发写入；且无备份机制。

**影响：** 数据丢失风险极高。

**解决方案：**
1. 迁移至 PostgreSQL（Prisma schema 已兼容，修改 `datasource db` 即可）
2. 配置定时备份（pg_dump + 存储到 OSS）
预计工作量：1 天。

## 四、上线优化项（P1 - 强烈建议）

> 不解决不会阻断上线，但影响用户体验或系统稳定性。

### P1-1：错误监控
**问题：** 无 Sentry 或类似错误追踪系统，线上报错无法及时感知。

**建议：** 集成 Sentry（前端 + 后端均需）。预计工作量：4 小时。

---

### P1-2：WebSocket 连接稳定性
**问题：** 当前 WebSocket 无重连机制校验，长时间网络切换后 socket 可能处于断开状态但前端未感知。

**建议：** 在前端 Socket.IO 客户端增加重连事件监听和 UI 提示（"连接断开，重新连接中..."）。预计工作量：4 小时。

---

### P1-3：图片上传大小与格式校验
**问题：** 当前后端对上传文件大小无校验（或限制过宽），存在大文件上传导致服务卡顿风险。

**建议：** 后端限制单文件 ≤ 10MB，前端上传前压缩（尤其是作品图）。预计工作量：4 小时。

---

### P1-4：API 限流（Rate Limiting）
**问题：** 当前仅在登录和验证码接口有速率限制，其他 API 无防刷机制。

**建议：** 为评论、消息发送、点赞等高频接口增加适当限流（NestJS ThrottlerGuard 已集成）。预计工作量：2-4 小时。

---

### P1-5：WebSocket CORS 收紧
**问题：** WebSocket Gateway 配置 `cors: { origin: '*' }`，生产环境应收紧到具体域名。

**建议：** 替换为 `origin: ['https://your-client-domain.com', 'https://your-tech-domain.com']`。预计工作量：30 分钟。

---

### P1-6：订阅功能完整闭环
**问题：** 美甲师端订阅页面已实现展示，但升级/变更订阅没有支付流程（仅展示计划），功能实际由超管后台手动操作。

**建议短期方案：** 美甲师端添加"联系客服升级"引导，明确告知当前为人工处理流程，避免用户困惑。预计工作量：2 小时。

---

### P1-7：Loading 状态与空状态完善
**当前问题点：**
- 部分页面在数据加载失败时仅 `console.error`，无用户可见的错误提示
- 某些列表页空状态文案比较简陋

**建议：** 统一处理网络错误（Toast 提示），完善空状态插图和文案。预计工作量：1 天。

## 七、技术债务汇总

| 项目 | 严重程度 | 备注 |
|------|---------|------|
| SQLite → PostgreSQL 迁移 | 🔴 严重 | 生产不可用 SQLite |
| 本地文件存储 → OSS | 🔴 严重 | 容器重启数据丢失 |
| 短信服务商未接入 | 🔴 严重 | 注册流程在生产中失效 |
| WebSocket CORS `*` | 🟡 中等 | 安全风险 |
| 验证码服务内存存储 | 🟡 中等 | 多实例部署时会失效 |
| 订单状态机有多余状态（pending_home/shop）| 🟡 中等 | 与前端 PRD 的 pending_quote→confirmed 不完全一致，需统一文档 |
| 部分页面错误仅 console.error | 🟢 轻微 | 影响用户体验 |
| 注释较少的业务逻辑 | 🟢 轻微 | 影响可维护性 |

## 八、总结

NailBook 是一个**架构完整、功能扎实**的美甲师私域运营 SaaS 系统。三端（客户端 / 美甲师端 / 超管后台）的核心业务闭环已全部实现，WebSocket 实时通信、完整的评论系统、RBAC 权限管理等技术能力均已落地。

**当前最大的风险不在于功能缺失，而在于基础设施层面**：SQLite 数据库、本地文件存储、短信服务商未接入这三点是生产上线的硬性阻塞，必须在上线前完成。

预计完成 P0 阻塞项后，可进入**正式测试 → 灰度上线**流程，总工时约 3-5 个工作日。
