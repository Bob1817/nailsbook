# 账号体系与登录注册流程设计

日期：2026-05-01

## 目标

建立一套统一、可验证的账号逻辑体系，覆盖超管后台、美甲师客户端、用户端三侧，使“获取验证码”“登录”“注册”“邀请码展示与分享”行为一致，并兼容当前开发态固定验证码 `123456`。

本次不接入真实短信服务，只在点击“获取验证码”时执行后端资格校验，校验通过后再允许进入固定验证码登录/注册流程。

## 范围

- 超管后台创建美甲师账号时生成唯一邀请码
- 美甲师端按手机号校验是否已被邀请
- 用户端区分登录获取验证码与注册获取验证码
- 用户端邀请注册与通用注册统一使用美甲师邀请码
- 美甲师端展示邀请码，并在分享链接中使用该邀请码

不在本次范围：

- 真实短信服务
- 多角色账号合并
- 重构现有 JWT 体系
- 重写全部旧测试，只补与本次行为直接相关的测试

## 关键收敛

`Technician.invitationCode` 收敛为唯一公开邀请码，承担以下职责：

- 美甲师唯一邀请 Id
- 美甲师端【我】页面展示值
- 用户注册时填写的邀请码
- 分享链接携带的邀请码
- 邀请注册页的默认邀请码

不再引入第二套“邀请 Id / 邀请码”字段，避免语义分裂。

## 账号状态模型

### 美甲师账号

- 来源：仅能由超管后台创建
- 标识：`Technician.phone`
- 邀请码：`Technician.invitationCode`
- 登录前提：
  - 该手机号存在对应 `Technician`
  - `Technician.status` 允许登录

### 用户账号

- 登录身份：`ClientUser`
- 与美甲师关系：`ClientTechBinding`
- 用户端“可登录”前提：
  - `ClientUser` 已存在
  - 至少有一条 `status = active` 的绑定

### 绑定关系

- 一个用户可绑定多个美甲师
- 一个用户可有一个默认美甲师
- 邀请注册或通用注册成功时，至少创建一条有效绑定

## 业务规则

### 1. 超管新增美甲师

- 超管后台新增美甲师时：
  - 校验手机号唯一
  - 生成唯一 `invitationCode`
  - 创建 `Technician`
- 创建成功后：
  - 该手机号立即具备美甲师端获取验证码登录资格
  - 美甲师详情页可查看邀请码

### 2. 美甲师端获取验证码

新增后端预检接口，前端点击“获取验证码”时先调用。

规则：

- 若手机号不存在对应 `Technician`：
  - 返回失败
  - 提示：`该账号未被邀请，无法登录`
- 若账号被禁用或非允许状态：
  - 返回失败
  - 提示沿用现有禁用语义
- 若通过校验：
  - 返回成功
  - 前端开始倒计时
  - 实际验证码仍为固定 `123456`

### 3. 美甲师端登录

- 登录接口仍使用手机号 + 开发验证码 `123456`
- 但前置体验依赖“获取验证码”资格校验
- 为避免绕过前端直接调用登录接口，后端登录接口也应保留同样的账号存在性和状态校验

### 4. 美甲师端【我】页与分享链接

- 【我】页面新增邀请码展示
- 分享链接改用 `Technician.invitationCode`
- 分享链接参数保持现有风格，但邀请码值统一来自 `invitationCode`

示例：

`/invite?invite_code=<Technician.invitationCode>`

如果现有分享链路还依赖 `tech_id`，可暂时保留 `tech_id` 作为辅助参数，但注册校验以 `invite_code` 为准。

### 5. 用户端通用登录页获取验证码

点击“获取验证码”时调用登录预检接口。

规则：

- 若手机号未注册：
  - 返回失败
  - 提示：`该手机号码还未注册，请先注册后再登录`
- 若手机号已注册但无任何有效绑定：
  - 返回失败
  - 提示：`该账号尚未绑定美甲师，请先通过邀请码注册/绑定`
- 若手机号已注册且存在有效绑定：
  - 返回成功
  - 前端开始倒计时
  - 实际验证码仍为固定 `123456`

### 6. 用户端邀请注册页

- 邀请链接中的 `invite_code` 直接作为本次注册邀请码
- 页面不要求用户再次手填邀请码
- 注册时校验邀请码有效性：
  - 无效则提示：`该邀请码无效，请跟您的美甲师确认后再注册`

注册行为：

- 若手机号未注册：
  - 创建 `ClientUser`
  - 创建与对应美甲师的 `ClientTechBinding`
  - 若是首个绑定，则设为默认绑定
- 若手机号已注册：
  - 若与该美甲师已有有效绑定，则按幂等成功处理
  - 若尚未绑定该美甲师，则新增一条有效绑定

### 7. 用户端通用注册

- 必填美甲师邀请码
- 点击“获取验证码”时走注册预检接口
- 注册预检要校验：
  - 手机号格式合法
  - 邀请码存在且有效
- 邀请码无效时提示：
  - `该邀请码无效，请跟您的美甲师确认后再注册`

通用注册提交时：

- 若手机号未注册，则创建用户并创建绑定
- 若手机号已注册：
  - 若已绑定该美甲师，按幂等成功处理
  - 若未绑定该美甲师，则补绑定

## 接口设计

### 美甲师端

新增：

- `POST /api/technician/auth/request-code`
  - 入参：`{ phone: string }`
  - 成功：`{ codeSent: true, devCode: "123456" }`
  - 失败：返回明确业务错误

现有：

- `POST /api/technician/auth/login`
  - 保留
  - 后端继续校验手机号存在与状态允许

### 用户端

新增：

- `POST /api/client/auth/request-login-code`
  - 入参：`{ phone: string }`
  - 仅已注册且有有效绑定时成功

- `POST /api/client/auth/request-register-code`
  - 入参：`{ phone: string, inviteCode: string }`
  - 用于邀请注册和通用注册的前置资格校验

现有保留：

- `GET /api/client/auth/find-by-invite-code`
- `POST /api/client/auth/register-by-invite`
- `POST /api/client/auth/login`

其中 `register-by-invite` 语义收敛为：

- 只认 `inviteCode`
- `techId` 可短期保留用于兼容旧前端，但服务端应以邀请码查找美甲师，避免“链接里 techId 对、邀请码不对”的不一致状态

## 前端调整

### 超管后台

- 创建美甲师无需新增字段
- 成功后在列表与详情继续展示邀请码
- 文案上将“邀请码”视为该美甲师唯一邀请 Id

### 美甲师端

- 登录页“获取验证码”改为先请求后端
- 仅请求成功后开始倒计时
- 错误提示展示后端返回文案
- 【我】页面显示邀请码
- 分享链接使用该邀请码

### 用户端

- 明确区分：
  - 登录模式
  - 通用注册模式
  - 邀请注册模式
- 登录模式“获取验证码”走 `request-login-code`
- 注册模式“获取验证码”走 `request-register-code`
- 邀请注册模式自动使用 URL 中的 `invite_code`
- 邀请注册模式不允许再输入其他邀请码
- 通用注册模式保留手填邀请码和边输边验能力

## 错误文案

以下文案属于本次约束的一部分，前后端保持一致：

- 美甲师未被邀请：`该账号未被邀请，无法登录`
- 用户未注册：`该手机号码还未注册，请先注册后再登录`
- 用户无有效绑定：`该账号尚未绑定美甲师，请先通过邀请码注册/绑定`
- 邀请码无效：`该邀请码无效，请跟您的美甲师确认后再注册`

## 数据与兼容性

- 不新增邀请码字段，继续使用 `Technician.invitationCode`
- 现有已创建的美甲师若邀请码缺失，需要在服务端补齐生成逻辑
- 现有用户若已注册但无绑定，登录资格应被拦截
- 多绑定结构保持不变，不回退为一对一模型

## 测试策略

优先补最窄测试，覆盖规则而不是铺满无关模块。

### 后端测试

- `TechniciansService.create`
  - 创建时生成唯一邀请码
- `TechnicianAuthService`
  - 已邀请手机号允许请求验证码
  - 未邀请手机号请求验证码失败并返回指定文案
- `ClientAuthService`
  - 登录预检：
    - 未注册失败
    - 已注册无绑定失败
    - 已注册且有绑定成功
  - 注册预检：
    - 邀请码无效失败
    - 邀请码有效成功
  - 邀请注册：
    - 未注册新建用户与绑定
    - 已注册同技师绑定幂等成功
    - 已注册未绑定该技师时新增绑定

### 前端验证

- 美甲师登录页只有在预检成功后才进入倒计时
- 用户通用登录页显示未注册文案
- 用户通用注册页显示无效邀请码文案
- 邀请注册页自动使用链接邀请码
- 美甲师【我】页展示邀请码

## 成功标准

- 超管新增美甲师后，该手机号可立即在美甲师端获取验证码并登录
- 未被超管创建的手机号，在美甲师端获取验证码时被拦截并提示未被邀请
- 美甲师端【我】页可见邀请码，分享链接使用同一邀请码
- 用户端未注册手机号在登录页获取验证码时被拦截并提示先注册
- 用户端通用注册必须依赖有效邀请码
- 邀请注册页可直接使用链接邀请码完成注册
- 已注册但无绑定的用户不能从通用登录入口获取验证码

## 风险与控制

### 风险 1：旧前端仍把“获取验证码”当纯本地行为

控制：

- 三端按钮统一改为“请求成功后再倒计时”

### 风险 2：`register-by-invite` 同时接受 `techId` 与 `inviteCode` 导致来源不一致

控制：

- 服务端优先以 `inviteCode` 查技师
- 若保留 `techId`，仅用于兼容旧调用，不作为最终权威依据

### 风险 3：已有测试仍基于旧的一对一假设

控制：

- 本次只修复与账号流程直接相关的测试
- 不做无关测试重构
